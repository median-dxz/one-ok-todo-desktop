# One Ok Todo 设计文档 V5

## 1. 应用定位与核心价值

### 1.1 应用定位

单人使用的轻量级时间线型任务管理工具。以"时间线"为组织单位管理任务，强调**逻辑依赖**与**展示顺序**的分离，降低用户心智负担。

### 1.2 核心设计理念

- **时间线即列表**：每条时间线是一个独立的任务序列，按用户定义的顺序展示
- **依赖隐式处理**：任务间的依赖关系作为后台逻辑约束，不体现在界面布局中
- **并行时间线**：多条时间线相互独立，可同时推进，互不干扰
- **备忘条功能**: 可视化 YAML 编辑器，支持条目切换类型与同级排序
- **多端同步**: 通过 WebDAV 实现数据同步

## 2. 数据模型与架构定义

### 2.1 引擎化架构设计

为了降低复杂状态计算与 UI 渲染间的耦合，项目采取了“状态引擎”与“布局引擎”分离的设计。全局 Store 承担数据容器与触发器的角色。

#### 2.1.1 状态计算引擎 (State Computation Engine)

设计独立的 `GraphEngine` 负责处理节点的级联更新逻辑。Store 作为“数据容器”和“触发器”。

- **纯函数设计**：输入旧的节点字典和操作，返回更新后的一批受影响节点。
- **状态触发**：以完成任务为例，Store 收到更新指令，调用引擎的 `updateStatus`，引擎遍历 `dependsOn` 或反向构建的图，计算出受影响的下游节点的增量更新（Diff），Store 再将 Diff 合并进 `state.nodes` 中。

#### 2.1.2 布局计算引擎 (Layout Engine)

在 `reactFlowStore` 和视图对象转换之间插入独立的 LayoutEngine 组件。

- **工作流**：AppStore 扁平数据 -> 抽取为基础 RFNode (无 position) -> `LayoutEngine` 计算坐标赋值 -> React Flow Store 渲染。
- **排版策略**：节点不依据连线排版，而是沿列表排列。对于支持时间轴的视图，Y 轴坐标由 Time -> Y 映射算出（如 `Y = (Date.now() - node.time) * 比例尺`）；X 轴用于并行处理互相独立的时间线互不重叠。

### 2.2 数据模型定义

#### 2.2.1 节点总体数据模型

当前实现以 `src/types/timeline.ts` 为准：节点分为 `TaskNode` 与 `DelimiterNode`，都继承自最基础的 `BaseNode`。

`milestone` 字段是用户手动设置的一个高亮标记，用于标识当前关注的重点任务或里程碑，它独立于任务本身的完成状态。

```typescript
// 基础节点接口
export interface BaseNode {
  id: string;          // 节点唯一标识 (nanoid)
  type: string;        // 节点类型 ('task' | 'delimiter')
  timelineId: string;  // 所属时间线 ID

  dependedBy: string[];// 被依赖的节点 ID 列表 (反向引用)
}

// 任务节点
export interface TaskNode extends BaseNode {
  type: 'task';
  title: string;       // 节点标题
  status: 'todo' | 'done' | 'skipped' | 'locked'; // 节点状态
  dependsOn: string[]; // 依赖的前置节点 ID 列表 (前向引用)
  content: {
    description: string;
    subtasks: Array<{ title: string; done: boolean }>;
  };
  milestone: boolean;        // 里程碑标记
  executionConfig?: object;  // 任务执行模式配置 (scheduled | quantitative)
  completedDate?: string;    // 完成/跳过的 ISO 时间字符串
}

// 分隔符节点
export interface DelimiterNode extends BaseNode {
  type: 'delimiter';
  markerType: 'start' | 'end' | 'date';
}

export type TimelineNode = TaskNode | DelimiterNode;
```

### 2.2.2 节点状态（NodeStatus）

节点状态统一交由**状态计算引擎（State Engine）**进行自动推导和更新判定。

| 状态      | 说明   | 业务规则                                              | 触发条件         |
| --------- | ------ | ----------------------------------------------------- | ---------------- |
| `todo`    | 待办   | 所有依赖项（dependsOn）都已完成或跳过，当前节点可执行 | **系统自动计算** |
| `done`    | 已完成 | 用户手动标记为完成                                    | 用户手动操作     |
| `skipped` | 已跳过 | 用户手动标记为跳过，视为一种特殊的`done`              | 用户手动操作     |
| `locked`  | 已锁定 | `dependsOn` 中存在至少一个未完成的前置项，不可执行    | **系统自动计算** |

### 2.2.3 任务执行模式（TaskMode）

这里定义了任务的两种执行配置（`ExecutionModeConfig`）：“定时模式”（scheduled）和“定量模式”（quantitative）。

**执行模式说明**:

- `scheduled`: 定时模式，任务到达指定时间前执行完成
- `quantitative`: 定量模式，需要完成指定数量才算完成

### 2.2.4 节点类型

`TimelineNode` 类型目前只包含 `TaskNode`，但数据结构上通过 `type` 字段（如 `'task'`, `'sub-task'`）的设计，保留了未来扩展不同节点类型的能力。`TaskNode` 是构成时间线的主体，它可以包含自己的依赖、子任务和各种配置。`Subtask` 是一种结构更简单的节点，用于 `TaskNode` 内部。

### 2.2.5 时间线结构

这里定义了“时间线”（Timeline）的构成。一条时间线包含标题、一系列按顺序排列的任务节点、以及这些节点间的依赖关系。

### 2.2.6 时间线分组

定义了“时间线分组”（TimelineGroup），它就像一个文件夹，可以把多条相关的“时间线”组织在一起，比如一个“工作”分组，一个“生活”分组。

时间线上的节点不能跨组添加依赖关系。

### 2.2.7 循环任务

循环任务是一种特殊的时间线，该时间线上面没有任务节点，而是根据当前的日期自动无限生成对应的节点，同时也记录了已经完成和跳过的任务，用于统计。

循环的种类有：天、周、月，每周的话可以指定周几，每月可以指定几号，还支持多个任务轮换执行（比如健身时“胸、背、腿”轮着来）。

同时支持将周/月任务按天均摊，均摊的可以是次数或者时间。

**循环任务特性**:

- **不包含 nodes 数组**: 循环任务的节点由时间计算生成，不持久化存储
- **统计信息**: 记录总完成次数、跳过次数等
- **任务轮换**: 支持 ABCABC 模式，每次循环使用不同的任务
- **时间生成逻辑**: 根据 `recurrence.frequency` 和时间参数自动计算当前周期的任务实例

## 3. 核心操作定义

### 3.1 组与时间线管理

#### 3.1.1 创建/编辑/删除/切换组

- [x] **创建组**：默认生成空组，追加到 `groupOrder` 数组末尾。
- [x] **编辑/删除组**：删除组时会级联删除其下的所有时间线和任务，并从 `groupOrder` 移除对应记录。
- [x] **切换当前组**：点击组列表可切换当前活动视图。

#### 3.1.2 调整组与时间线的展示顺序

- [x] **交换组顺序**：通过拖拽或按钮上下移动组位置，实质为修改平移 `groupOrder` 数组中 ID 顺序，不影响内部数据归属。
- [] **时间线管理**：在当前组内创建（追加到组的 `timelineIds` 数组末尾）、重命名或删除时间线（级联删除旗下任务）。

### 3.2 节点基础操作

#### 3.2.1 创建节点

- [] 新任务节点可直接添加至对应 Timeline 的尾部（追加至 `nodeIds` 序列）；新创建的节点默认无依赖（`dependsOn: []`），可后续配置。

#### 3.2.2 删除节点

- [] 删除单一节点，从所属 `nodeIds` 中移出并清空其本身数据。这不触发连带删除。若其他节点 `dependsOn` 中依赖了此节点，则保留孤儿依赖（计算时视同已满足）。

#### 3.2.3 调整节点顺序

用户只能在**同一时间线内**拖拽或执行位移修改 `nodeIds` 数组内部排序。不支持跨时间线的列表乱移。

#### 3.2.4 编辑节点

用户可以修改节点的标题、执行模式、截止时间以及子任务列表等内容属性。

#### 3.2.5 完成/跳过节点

用户可以将节点状态设为 `done` / `skipped`。该操作将触发状态引擎（State Engine）重新计算，更新级联下游节点状态使其解锁为 `todo`。

#### 3.2.6 撤销完成

核心规则：只有当节点的所有**直接后继节点**（其他在其 `dependsOn` 中包含自身的节点）状态都是 `todo` 或 `locked` 时，才允许撤回其完成/跳过状态。如果任何直接后继节点已发生状态突变（变为了 `done` / `skipped`），那么拦截撤销操作以防止依赖链出现矛盾。

### 3.3 依赖管理操作

#### 3.3.1 添加依赖

- 用户在任务编辑界面手动添加目标依赖对象至 `dependsOn` 中。
- 引擎实时校验 DAG，避免触发循环依赖。
- 一旦挂载新依赖，会重新计算判定当前节点是否须进入锁定（`locked`）。

#### 3.3.2 移除依赖

- 摘除某项依赖关联，重新计算当前节点状态（可能由 `locked` 解锁为 `todo`）。

### 3.4 核心业务规则约束

#### 3.4.1 任务节点状态计算

**状态计算规则**：

- `locked`: `dependsOn` 中存在任一节点不是 `done`/`skipped`
- `todo`: `dependsOn` 为空或全部为 `done`/`skipped`
- `done` / `skipped`: 用户手动设置

**触发时机**：

- 创建节点时：新节点默认无依赖，状态为 `todo`
- 编辑依赖时：添加或移除依赖后重新计算状态
- 完成/跳过/撤销节点时：更新状态后触发依赖检查，更新下游节点状态

#### 关于节点顺序的约束

**交换节点顺序**:

- 只能在同一时间线内调整节点顺序，禁止跨时间线移动。
- 调整顺序不影响节点间的依赖关系，依赖关系由 `dependsOn` 定义，与视觉顺序无关。
- 调整顺序时对于未完成的节点，必须要满足截止日期的先后约束；对于已完成的节点，必须要满足完成日期的先后约束。
- 如果撤销节点完成状态后/修改截止日期违反了约束，则自动插入该节点到最近的一个合法位置。
- 可以使用自动整理功能排序节点，自动整理会按照完成日期和截止日期排序节点。

## 4. 数据持久化与同步

### 4.1 存储格式

- **当前实现**：Zustand `persist` 通过 `src/utils/storage.ts` 保存一份“根数据”（包含 `appMetadata` / `memo` / `timelineGroups`），并使用 `superjson` 序列化。
  - Tauri：Rust 侧文件路径为 `Documents/OneOkTodo/data/root-data.json`（实现见 `src-tauri/src/lib.rs`）
  - Web：IndexedDB（`idb`）

> 说明：文档 v4 里提到的“多文件拆分（timelines.json / memos.json / metadata.json）”在当前代码中未落地。

### 4.2 WebDAV 同步策略

- **当前实现状态**：WebDAV 作为持久化适配器接入聚合层（见 `src/utils/storage.ts`），与本地（Tauri 文件）和 Web（IndexedDB）一起参与读写。
- **读取策略**：应用启动或手动 Sync 时通过 `persist.rehydrate()` 触发聚合读取，按“最新且非空文档”选主（暂不处理复杂冲突合并）。
- **写入策略**：状态持久化时向可用数据源广播写入，WebDAV 传输统一使用 `superjson`。

### 4.3 多数据源聚合层（规划）

- 在持久化层之上增加“适配器聚合层”，统一接入：
  - 本地数据源（Tauri/Rust 文件）
  - 纯 Web 数据源（IndexedDB）
  - 远端数据源（WebDAV）
- 启动时并发读取多个数据源，按“最新且非空文档”选主（暂不处理复杂冲突合并）。
- 保存时向可用数据源广播写入，保证本地与远端尽可能保持一致。

### 4.4 Schema 演进与容错策略（规划）

- 数据读取后先经过 Zod 校验与恢复流程。
- **可恢复错误**：如字段缺失但可提供默认值、旧字段可映射到新结构，允许通过 Zod 进行恢复后继续加载。
- **不可恢复错误**：如关键结构断裂、无法判定语义的类型破坏，必须抛出错误兜底。
- 抛错时要求双通道反馈：
  - 控制台输出完整错误信息（便于排查）
  - UI 通过 Toaster 提示用户“数据不兼容/已损坏，当前源加载失败”
- 禁止静默吞错导致无提示的数据丢失或错误覆盖。

## 5. 备忘条功能模块 (Memo Module)

### 5.1 功能定位

- 当前实现为一个独立于 Timeline 的“树状结构化备忘录”，用于记录层级信息。
- UI 入口：`src/components/memo/MemoDisplay.tsx`，节点组件：`src/components/memo/MemoNode.tsx`。
- 数据与 Timeline 一样通过 `useAppStore` 持久化（字段为 `memo: MemoNode[]`，类型在 `src/types/memo.ts`）。

### 5.2 数据模型

- 节点类型：`'string' | 'object' | 'array' | 'number' | 'boolean'`，并用 `children: MemoNode[]` 表达树结构。
- 当前 UI 主要支持编辑节点的 `key`（见 `Editable`），value 编辑/搜索/拖拽排序尚未完整实现。

### 5.3 核心操作

- **可视化编辑**：新增根节点/子节点（选择类型对话框）、删除节点、编辑节点 key。
- 具体 action 位于 `src/store/memoSlice.ts`（如 `addMemoNode` / `deleteMemoNode` / `updateMemoNode`）。

### 5.4 UI/UX 需求

- 当前 UI 已具备树状视图与内联操作（新增/删除/编辑 key）。
- 折叠、拖拽排序、类型切换与 value 编辑属于后续可扩展项，当前代码未完整覆盖。
