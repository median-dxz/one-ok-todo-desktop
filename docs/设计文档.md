# One Ok Todo 设计文档 V4

## 1. 应用定位与核心价值

### 1.1 应用定位

单人使用的时间线型任务管理工具，以"时间线思维"组织任务，而非传统的列表或看板模式。

### 1.2 核心特性

- **时间线优先**: 任务以时间线（Timeline）为组织单位，体现事件的顺序性
- **多维依赖关系**: 支持节点间依赖、跨 Timeline 依赖、多分支依赖
- **灵活循环任务**: 支持多种循环模式、任务轮换、定时/定量执行
- **备忘条功能**: 可视化 YAML 编辑器，支持条目切换类型与同级排序
- **多端同步**: 通过 WebDAV 实现数据同步

## 2. 数据模型定义

### 2.2.1 节点总体数据模型

当前实现以 `src/types/timeline.ts` 为准：节点分为 `TaskNode` 与 `DelimiterNode`，都继承自最基础的 `BaseNode`（依赖关系在 base 上）。

```typescript
// 基础节点属性（当前实现）
export type BaseNode = {
  id: string;
  type: 'task' | 'delimiter';
  prevs: string[];
  succs: string[];
};

export type TaskNode = BaseNode & {
  type: 'task';
  title: string;
  description?: string;
  status: NodeStatus;
  executionConfig?: ExecutionModeConfig;
  subtasks?: SubTask[];
  milestone?: boolean;
  completedDate?: string;
};

export type DelimiterNode = BaseNode & {
  type: 'delimiter';
  markerType: 'start' | 'end';
};

export interface SubTask {
  title: string;
  status: 'todo' | 'done';
}
```

`milestone` 字段是用户手动设置的一个高亮标记，用于标识当前关注的重点任务或里程碑，它独立于任务本身的完成状态。

### 2.2.2 节点状态（NodeStatus）

节点状态枚举以代码为准（`src/types/timeline.ts` / `src/types/schemas.ts`）。
注意：当前实现里 **并没有** 全局的“根据依赖自动派生 lock/todo”的统一逻辑；状态主要由 UI 操作直接写入（见 `src/store/reactFlowStore.ts` 的 `updateTaskNodeStatus`）。

| 状态      | 说明   | 业务规则                                 | 触发条件         |
| --------- | ------ | ---------------------------------------- | ---------------- |
| `todo`    | 待办   | 所有前驱节点都已完成，当前节点可执行     | **系统自动计算** |
| `done`    | 已完成 | 用户手动标记为完成                       | 用户手动操作     |
| `skipped` | 已跳过 | 用户手动标记为跳过，视为一种特殊的`done` | 用户手动操作     |
| `lock`    | 已锁定 | 至少有一个前驱节点未完成，不可执行       | （规划）         |

（规划中的）状态计算规则（与最初设计一致，但当前代码未实现自动重算）：
- `lock`: `prevs` 中存在任一节点不是 `done`/`skipped`
- `todo`: `prevs` 为空或全部为 `done`/`skipped`
- `done` / `skipped`: 用户手动设置

### 2.2.3 任务执行模式（TaskMode）

这里定义了任务的两种执行配置（`ExecutionModeConfig`）：“定时模式”（scheduled）和“定量模式”（quantitative）。

**执行模式说明**:

- `scheduled`: 定时模式，任务到达指定时间前执行完成
- `quantitative`: 定量模式，需要完成指定数量才算完成

### 2.2.4 节点类型

`TimelineNode` 类型目前只包含 `TaskNode`，但数据结构上通过 `type` 字段（如 `'task'`, `'sub-task'`）的设计，保留了未来扩展不同节点类型的能力。`TaskNode` 是构成时间线的主体，它可以包含自己的依赖、子任务和各种配置。`SubTask` 是一种结构更简单的节点，用于 `TaskNode` 内部。

### 2.2.5 时间线结构

这里定义了“时间线”（Timeline）的构成。一条时间线包含标题、一系列按顺序排列的任务节点、以及这些节点间的依赖关系。

### 2.2.6 时间线分组

定义了“时间线分组”（TimelineGroup），它就像一个文件夹，可以把多条相关的“时间线”组织在一起，比如一个“工作”分组，一个“生活”分组。

时间线上的节点不能跨组添加依赖关系。

### 2.2.7 循环任务

循环任务是一种特殊的时间线，该时间线上面没有任务节点，而是根据当前的日期自动无限生成对应的节点，同时也记录了已经完成和跳过的任务，用于统计。

循环的种类有：天、周、月，每周的话可以指定周几，每月可以指定几号，还支持多个任务轮换执行（比如健身时“胸、背、腿”轮着来）。

同时支持将周/月任务按天均摊，均摊的可以是次数或者时间。

**循环任务特性**:

- **不包含 nodes 数组**: 循环任务的节点由时间计算生成，不持久化存储
- **统计信息**: 记录总完成次数、跳过次数等
- **任务轮换**: 支持 ABCABC 模式，每次循环使用不同的任务
- **时间生成逻辑**: 根据 `recurrence.frequency` 和时间参数自动计算当前周期的任务实例

## 3. 核心操作定义

### 3.1 时间线操作

#### 3.1.1 创建时间线

用户可以创建一个新的时间线，新创建的时间线是空的，不附带任何初始节点。

#### 3.1.2 编辑时间线

用户可以修改已有时间线的标题等基本信息。

#### 3.1.3 删除时间线

用户可以删除一个时间线。删除时，会一并移除其中包含的所有节点和相关的依赖关系。

### 3.2 分组操作

#### 3.2.1 创建/编辑/删除分组

用户可以创建新的时间线分组（文件夹），或编辑已有分组的名称。

### 3.3 节点基础操作

#### 3.3.1 创建节点

（当前实现）用户可以在时间线内创建新任务节点，插入方式是“相对某个已选节点插到前面/后面”，由 UI 传入 `insertMode: 'prev' | 'succ'` 完成（见 `src/store/reactFlowStore.ts` 的 `addTaskNode`）。

（规划）更通用的“任意指定 prevs/succs 精确插入”在当前代码中未完整实现。

#### 3.3.2 删除节点

（规划）删除节点并自动重连依赖（如 `A -> B -> C` 删除 B 后变为 `A -> C`）在当前代码中未实现。

#### 3.3.3 编辑节点

用户可以修改节点的标题、执行模式、子任务列表等内容属性。

#### 3.3.4 完成/跳过节点

（当前实现）用户可以将节点状态改为 `done` / `skipped` / `todo`（见 `src/store/reactFlowStore.ts` 的 `updateTaskNodeStatus`）。

（规划）“触发后继节点状态自动重新计算”尚未形成统一实现。

#### 3.3.5 撤销完成

（当前实现）Undo Completion 直接将 `done` / `skipped` 改回 `todo`，未做依赖完整性校验。

（规划）可在未来加入严格条件校验（例如前驱必须 `done`/`skipped`，后继必须 `todo`/`lock`）。

### 3.4 节点高级图操作

#### 3.4.1 添加依赖（Merge）

（规划）在两个节点之间创建新的依赖关系（Merge）当前未实现。Right Panel 中存在按钮占位（见 `src/components/timeline/RightPanel.tsx`）。

- 两个节点的状态都必须是 `todo` 或 `lock`。
- 两个节点处于同一个 Timeline 组。
- 添加新的依赖关系后，整个项目的所有节点必须依然构成一个有向无环图（DAG），不允许出现循环依赖。

#### 3.4.2 创建并行分支（Fork）

（规划）Fork 逻辑当前未实现（按钮占位）。

1. 在源节点之后创建一个新的后继节点。
2. 创建一条全新的时间线。
3. 将这个新创建的后继节点作为新时间线的起始节点。

#### 3.4.3 回滚节点（Revert）

（规划）Revert 逻辑当前未实现（按钮占位）。

- 如果该时间线的最后一个节点存在一个位于**其他时间线**的后继节点，并且该后继节点也已完成（`done`/`skipped`），则不允许执行 Revert 操作，以防止破坏已完成的跨时间线依赖链。

#### 3.4.4 交换节点内容

（规划）Swap 逻辑当前未实现（按钮占位）。

- 两个节点的状态都必须是 `todo` 或 `lock`。
- 操作仅交换节点的“标签数据”（如标题、描述、子任务列表等），节点的 `id`、`prevs` 和 `succs` 保持不变。

## 4. 数据持久化与同步

### 4.1 存储格式

- **当前实现**：Zustand `persist` 通过 `src/utils/storage.ts` 保存一份“根数据”（包含 `appMetadata` / `memo` / `timelineGroups`），并使用 `superjson` 序列化。
  - Tauri：Rust 侧文件路径为 `Documents/OneOkTodo/data/root-data.json`（实现见 `src-tauri/src/lib.rs`）
  - Web：IndexedDB（`idb`）

> 说明：文档 v4 里提到的“多文件拆分（timelines.json / memos.json / metadata.json）”在当前代码中未落地。

### 4.2 WebDAV 同步策略

- **当前实现状态**：WebDAV 作为持久化适配器接入聚合层（见 `src/utils/storage.ts`），与本地（Tauri 文件）和 Web（IndexedDB）一起参与读写。
- **读取策略**：应用启动或手动 Sync 时通过 `persist.rehydrate()` 触发聚合读取，按“最新且非空文档”选主（暂不处理复杂冲突合并）。
- **写入策略**：状态持久化时向可用数据源广播写入，WebDAV 传输统一使用 `superjson`。

### 4.3 多数据源聚合层（规划）

- 在持久化层之上增加“适配器聚合层”，统一接入：
  - 本地数据源（Tauri/Rust 文件）
  - 纯 Web 数据源（IndexedDB）
  - 远端数据源（WebDAV）
- 启动时并发读取多个数据源，按“最新且非空文档”选主（暂不处理复杂冲突合并）。
- 保存时向可用数据源广播写入，保证本地与远端尽可能保持一致。

### 4.4 Schema 演进与容错策略（规划）

- 数据读取后先经过 Zod 校验与恢复流程。
- **可恢复错误**：如字段缺失但可提供默认值、旧字段可映射到新结构，允许通过 Zod 进行恢复后继续加载。
- **不可恢复错误**：如关键结构断裂、无法判定语义的类型破坏，必须抛出错误兜底。
- 抛错时要求双通道反馈：
  - 控制台输出完整错误信息（便于排查）
  - UI 通过 Toaster 提示用户“数据不兼容/已损坏，当前源加载失败”
- 禁止静默吞错导致无提示的数据丢失或错误覆盖。

## 5. 备忘条功能模块 (Memo Module)

### 5.1 功能定位

- 当前实现为一个独立于 Timeline 的“树状结构化备忘录”，用于记录层级信息。
- UI 入口：`src/components/memo/MemoDisplay.tsx`，节点组件：`src/components/memo/MemoNode.tsx`。
- 数据与 Timeline 一样通过 `useAppStore` 持久化（字段为 `memo: MemoNode[]`，类型在 `src/types/memo.ts`）。

### 5.2 数据模型

- 节点类型：`'string' | 'object' | 'array' | 'number' | 'boolean'`，并用 `children: MemoNode[]` 表达树结构。
- 当前 UI 主要支持编辑节点的 `key`（见 `Editable`），value 编辑/搜索/拖拽排序尚未完整实现。

### 5.3 核心操作

- **可视化编辑**：新增根节点/子节点（选择类型对话框）、删除节点、编辑节点 key。
- 具体 action 位于 `src/store/memoSlice.ts`（如 `addMemoNode` / `deleteMemoNode` / `updateMemoNode`）。

### 5.4 UI/UX 需求

- 当前 UI 已具备树状视图与内联操作（新增/删除/编辑 key）。
- 折叠、拖拽排序、类型切换与 value 编辑属于后续可扩展项，当前代码未完整覆盖。
